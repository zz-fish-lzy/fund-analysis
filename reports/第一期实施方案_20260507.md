## 一、项目背景与目标

### 1.1 背景

部署报告（2026-05-07）完成了三项基础设施搭建：
- TrendRadar 热点监控（GitHub Actions，每天 8:00/16:00）
- Supabase 云数据库（6 张表，PostgreSQL 17）
- 基金项目 database.py 集成（支持云端读写）

但这些基础设施尚未产生实际价值 — TrendRadar 的新闻数据存在 COS 中未同步到数据库，基金数据仍在本地 CSV，模型训练结果未入库。

### 1.2 第一期目标

将基础设施转化为可用的数据分析流水线，具体包括：

| 模块 | 目标 | 产出 |
|------|------|------|
| TrendRadar 数据同步 | 新闻从 COS → Supabase，经过滤、分类、量化 | `trend_news` 表（带情感/分类/影响力） |
| 基金数据入库 | AKShare 数据经筛选后写入 Supabase | `funds` + `fund_nav` + `fund_scores` 表 |
| 模型训练集成 | 训练结果和预测写入数据库 | `predictions` 表（含评估指标） |
| 情感分析流水线 | DeepSeek API → 本地分类器蒸馏 | 零成本自动化情感分析能力 |

## 二、TrendRadar 数据同步（新闻流水线）

### 2.1 设计思路

全量新闻直接喂模型是量化投资最常见的错误 — 90% 以上的新闻对基金价格没有显著影响。设计四步流水线：

**源头过滤 → 内容过滤 → 分类量化 → 精准关联**

### 2.2 第一步：源头过滤

#### RSS 源配置

在 TrendRadar 的 `config/config.yaml` 中添加 RSS 源（2026 年 5 月验证可用）：

| 类别 | 源名称 | 更新频率 | 说明 |
|------|--------|----------|------|
| 宏观经济 | 财联社 7x24 滚动 | 5-10 分钟 | 国内最快财经快讯 |
| 宏观经济 | 第一财经 财经要闻 | 30 分钟 | 深度报道和宏观分析 |
| 宏观经济 | 证券时报 要闻 | 1 小时 | 权威政策解读 |
| 宏观经济 | 中国证券报 要闻 | 1 小时 | 官方背景，可靠性最高 |
| 行业-科技 | 财联社-科技 | - | 对应科技主题基金 |
| 行业-新能源 | 财联社-新能源 | - | 对应新能源主题基金 |
| 行业-医药 | 财联社-医药 | - | 对应医药主题基金 |
| 行业-消费 | 财联社-消费 | - | 对应消费主题基金 |
| 行业-金融 | 财联社-金融 | - | 对应金融主题基金 |
| 基金专属 | 东方财富-基金要闻 | 2 小时 | 专门的基金行业新闻 |
| 债券专属 | 中国债券信息网 | - | 跟踪债券基金时使用 |

**已失效/不推荐的源：** 新浪财经 RSS（2025 年底全部停更）、同花顺 RSS（格式不规范）、财新网 RSS（需付费订阅）。

#### TrendRadar 内置爬虫补充

TrendRadar 本身内置了财联社和华尔街见闻的热榜爬虫，在 `config.yaml` 的 `platforms` 部分配置即可，无需额外 RSS。

#### 关键词过滤

在 `frequency_words.txt` 中添加基金相关关键词：

```
基金、净值、分红、限购、基金经理、持仓
沪深300、中证500、创业板、科创板
新能源、医药、消费、半导体、人工智能
降息、降准、加息、CPI、PMI、GDP
```

#### 必须屏蔽的垃圾源

删除 TrendRadar 默认的"微博热搜""知乎热榜""娱乐""体育"等板块，99.9% 与基金无关。

### 2.3 第二步：内容过滤

即使是财经新闻也有大量噪音（公司招聘、行业会议通知等）。用关键词白名单+黑名单规则快速过滤：

**白名单**（包含才保留）：降息、加息、降准、GDP、CPI、PMI、沪深300、中证500、新能源、医药、消费、半导体、人工智能、政策、利好、利空、基金、持仓、净值、分红、限购、基金经理

**黑名单**（包含直接丢弃）：招聘、年会、辟谣、致歉、离职、任命、专访、直播、广告、推广、开户、荐股、骗局、维权

### 2.4 第三步：新闻分类与影响量化

#### 方案：DeepSeek API 零样本分类

不自己训练情感分析模型（通用情感模型对金融文本完全没用），直接调用 DeepSeek API，效果远超任何开源小模型。

**Prompt 模板：**
```
你是一个专业的金融新闻分析师，请分析以下新闻，严格按照JSON格式输出结果：
1. category：新闻分类，只能从以下选项中选一个：
   ["宏观经济", "新能源", "医药生物", "消费", "科技", "金融地产", "债券", "其他"]
2. sentiment：情感倾向，-1（利空）、0（中性）、1（利好）
3. impact：影响程度，1-5分（1=影响极小，5=影响极大）
4. reason：简短说明原因（不超过20字）

新闻标题：{news_title}
新闻内容：{news_content}
```

#### 新闻来源权重

不同来源的新闻重要性不同：

| 来源 | 权重 | 说明 |
|------|------|------|
| 新华社 / 人民日报 / 央行官网 | 2.0 | 最高权威性 |
| 财新 / 第一财经 / 新浪财经 | 1.0 | 主流财经媒体 |
| 自媒体 / 股吧 | 0.2 | 噪声大，甚至可直接过滤 |

#### 时间衰减系数

新闻影响随时间减弱：

| 时间 | 衰减系数 |
|------|----------|
| 当天 | 1.0 |
| 第 2 天 | 0.5 |
| 第 3 天 | 0.2 |
| 3 天以上 | 0 |

### 2.5 第四步：新闻与基金精准关联

#### 基金-新闻类别映射规则

| 基金类型 | 关联的新闻类别 |
|----------|---------------|
| 沪深300 / 中证500 指数基金 | 宏观经济 + 所有行业 |
| 新能源主题基金 | 新能源 + 宏观经济 |
| 医药主题基金 | 医药生物 + 宏观经济 |
| 纯债基金 | 宏观经济（利率/通胀）+ 债券 |
| 偏股混合基金 | 前三大重仓行业 + 宏观经济 |

#### 每日事件影响分计算

```
基金当日事件分 = Σ(新闻情感 × 新闻影响程度 × 新闻权重 × 时间衰减系数)
```

### 2.6 调度时间安排

#### 交易日（周一至周五）

| 环节 | 时间 | 说明 |
|------|------|------|
| TrendRadar 爬取 | 8:00 / 16:00 | 保持原样 — 覆盖早盘/午盘后新闻更新高峰，匹配 RSS 源 5-10 分钟更新频率 |
| 数据处理 | 18:00 后 | 避开 AKShare 交易时段限流（9:30-15:00），且覆盖 16:00 最后一批爬取数据 |
| 邮箱通报 | 19:00-20:00 | 预留 1 小时完成新闻筛选→API 标注→基金数据整合，确保推送"有效信息" |

#### 非交易日（周末）

| 环节 | 时间 | 说明 |
|------|------|------|
| TrendRadar 爬取 | 8:00 / 16:00 | 保持不变，周末仍有宏观政策/行业新闻 |
| 数据处理 + 通报 | 合并为 18:30 一次 | 非交易日新闻量少，无需分两次处理，降低冗余推送 |

#### 法定节假日

| 环节 | 时间 | 说明 |
|------|------|------|
| TrendRadar 爬取 | 仅 10:00 一次 | 仅覆盖核心宏观新闻，减少无效爬取 |
| 数据处理 + 通报 | 合并为 11:00 一次 | 节假日新闻量极少，单次处理即可 |

#### 分层推送策略

将邮箱推送分为两档，避免信息过载：

| 推送档位 | 时间 | 内容 |
|----------|------|------|
| 早间轻量推送 | 10:00（基于 8:00 爬取数据） | 仅含宏观经济重大新闻（如降息/降准）+ 重点基金早间净值 |
| 晚间完整推送 | 19:00（基于 8:00+16:00 全量数据） | 所有分类新闻 + 基金当日评分 + 次日预测摘要 |

> **注**：早间轻量推送为可选优化项，第一期可先只做晚间完整推送，后续根据实际需求添加。

#### 失败告警机制

若数据处理或推送流程失败，立即触发邮箱告警（同一推送邮箱），避免流程中断未察觉。告警内容包括：失败环节、错误信息、建议处理方式。

### 2.7 避坑指南

1. **绝对不要用通用情感分析模型**（如 snownlp）— 它会把"央行加息"判断为中性
2. **不要过度依赖新闻** — 新闻只是辅助因子，建议在模型中权重不超过 30%
3. **避免前视偏差** — 回测时只能用 T 日及之前的新闻预测 T+1 日净值
4. **定期更新白名单和黑名单** — 每年更新一次，加入新热点行业

## 三、基金数据入库（筛选 + 动态评分）

### 3.1 设计思路

全量入库（1 万+ 只基金）浪费存储和计算资源，还会被"垃圾基金"干扰。采用三级方案：**硬指标初筛 → 动态评分池 → 分层存储**，将入库量控制在 100-200 只。

### 3.2 第一级：硬指标初筛（过滤 90% 基金）

#### 一票否决指标

| 指标 | 筛选阈值 | 原因 |
|------|----------|------|
| 基金规模 | ≥2 亿，≤200 亿 | <2 亿清盘风险高，>200 亿超额收益难 |
| 成立时间 | ≥1 年 | 至少有一轮牛熊/风格切换的历史数据 |
| 基金经理任职时间 | ≥1 年 | 避免频繁更换导致风格漂移 |
| 日成交额 | ≥100 万 | 流动性好，大额申赎不会冲击净值 |
| 费率 | 管理费≤1.5%，托管费≤0.25% | 过高费率长期侵蚀收益 |

#### 加分指标（满足 ≥2 项才入库）

| 指标 | 优秀阈值 | 原因 |
|------|----------|------|
| 近 1 年年化收益率 | 超越同类平均≥5% | 历史业绩优秀 |
| 近 1 年最大回撤 | ≤同类平均的 80% | 风控能力强 |
| 机构持仓比例 | ≥20%，≤80% | 机构认可且不被单一机构控盘 |
| Sharpe 比率 | ≥1.0（近 1 年） | 风险调整后收益高 |

### 3.3 第二级：动态评分池（保证只留潜力基金）

#### 评分体系（满分 100 分）

| 维度 | 权重 | 评分规则 |
|------|------|----------|
| 业绩表现 | 40 分 | 近1月10分 + 近3月15分 + 近1年15分（按同类排名分档） |
| 风险控制 | 25 分 | 最大回撤15分 + 波动率10分（越小分越高） |
| 基金经理 | 20 分 | 任职时间10分 + 管理的其他基金业绩10分 |
| 资金动向 | 15 分 | 机构持仓变化8分 + 基金规模变化7分（正向变化加分） |

#### 动态管理规则

- **每周更新**：重新计算所有入库基金的评分
- **重点池**（评分前 100）：模型训练和推荐的核心
- **观察池**（评分 101-200）：保留数据但不参与训练
- **淘汰池**（评分 200+）：标记为"非重点"，停止更新净值（保留历史数据用于回测）
- **每月纳新**：用初筛规则扫描全市场，把新符合条件的基金加入观察池

### 3.4 第三级：Supabase 分层存储优化

利用 PostgreSQL 特性设计表结构和索引，查询速度提升 10 倍以上。

#### 查询优化技巧

- **模型训练只查重点池**：通过 `is_focus = TRUE` 过滤
- **建视图**：创建 `focus_funds_latest` 视图，一键获取重点基金最新数据
- **索引优化**：在 `fund_code`、`nav_date`、`is_focus`、`pool_type` 上建索引

### 3.5 AKShare 限流策略

#### 实测安全阈值（2026 年 5 月）

| 接口类型 | 安全阈值 | 危险阈值 | 触发后果 |
|----------|----------|----------|----------|
| 单只基金接口 | 15 次/分钟 | >20 次/分钟 | 先返回空数据，持续高频封 IP 1-24 小时 |
| 批量接口 | 1 次/5 分钟 | >3 次/10 分钟 | 临时限流 10 分钟 |
| 持仓/财报接口 | 10 次/分钟 | >15 次/分钟 | 直接返回 403 |

**关键注意：** 数据中心 IP（GitHub Actions、云服务器）的阈值只有家庭宽带的 30%。批量数据获取任务必须在本地家庭网络运行，且必须在 18:00 后执行（详见 2.6 调度时间安排）。

#### 最优请求间隔

使用随机抖动的动态间隔（固定节奏是反爬系统最容易识别的特征）：

- 单只基金接口：基础间隔 2.5 秒 + 0.5-1.5 秒随机抖动
- 持仓/财报接口：基础间隔 4 秒 + 1-2 秒随机抖动

#### 分批处理方案

| 阶段 | 每日处理量 | 分批方式 | 总耗时 |
|------|-----------|----------|--------|
| 首次全量初筛 | 200 只/天 | 分 4 批，每批 50 只，批间休息 10 分钟 | 5 天完成 1000 只 |
| 每日净值更新 | 全量重点池（100 只） | 分 2 批，每批 50 只，批间休息 5 分钟 | 10 分钟完成 |
| 每周评分更新 | 全量观察池（200 只） | 分 4 批，每批 50 只，批间休息 10 分钟 | 1 小时完成 |

#### 额外避坑技巧

- **异常处理与重试**：实现指数退避重试机制，最多重试 3 次
- **避免高峰时段**：不要在交易日 9:30-15:00 运行批量任务，建议 18:00 后或周末
- **本地缓存**：使用 `diskcache` 缓存已获取的数据，避免重复请求
- **优先使用批量接口**：`ak.fund_name_em()` 一次获取 1 万+ 只基金列表，`ak.fund_open_fund_daily_em()` 一次获取全市场当日净值

### 3.6 避坑指南

1. **初筛不要太严格** — 先宽松入库，靠动态评分淘汰，避免错过潜力黑马
2. **保留历史数据** — 即使基金被标记为"非重点"，也不要删除历史净值数据（回测需要）
3. **分批处理** — 全市场 1 万+ 只基金不要一次性跑完
4. **定期更新初筛规则** — 每年调整一次规模阈值

## 四、模型训练集成

### 4.1 核心评估指标体系

基金预测的核心不是"预测准确率"，而是"风险调整后收益"。从三个维度综合评估：

#### 第一维度：预测性能指标（基础参考）

| 指标 | 含义 | 目标值 |
|------|------|--------|
| MAE | 平均绝对误差 | 越小越好 |
| RMSE | 均方根误差（对大误差更敏感） | 越小越好 |
| MAPE | 平均绝对百分比误差 | <10% 优秀 |
| 方向准确率 | 预测涨跌方向的正确率 | >55% 可用，>60% 优秀 |

#### 第二维度：回测收益指标（核心依据）

| 指标 | 含义 | 目标值 |
|------|------|--------|
| 累计收益率 | 策略在整个回测期的总收益 | 越高越好 |
| 年化收益率 | 折算到每年的收益率 | 股票型>15%，混合型>10%，债券型>5% |
| 超额收益率 | 相对于基准的额外收益 | 越高越好（证明有 Alpha） |
| 信息比率 (IR) | 超额收益的稳定性 | >0.5 可用，>1.0 优秀 |

#### 第三维度：风险控制指标（保命线）

| 指标 | 含义 | 目标值 |
|------|------|--------|
| 最大回撤 (MDD) | 从最高点到最低点的最大跌幅 | 股票型<30%，混合型<20%，债券型<5% |
| 夏普比率 (Sharpe) | 承担单位风险获得的超额收益 | >1.0 可用，>1.5 优秀，>2.0 卓越 |
| 卡尔马比率 (Calmar) | 年化收益率与最大回撤的比值 | >0.5 可用，>1.0 优秀 |
| 索提诺比率 (Sortino) | 只考虑下行风险（更适合基金） | >1.0 可用，>1.5 优秀 |

### 4.2 框架选择策略

按复杂度递增，分阶段实施：

| 阶段 | 框架 | 适用场景 | 项目状态 |
|------|------|----------|----------|
| 第一步 | XGBoost/LightGBM + Backtrader | 跑通最小原型 | 已有 `src/model_lgbm.py` |
| 第二步 | 微软 Qlib | 专业量化回测 | 已有 `qlib/` 目录 |
| 第三步 | FinRL 强化学习 | 反馈迭代闭环 | 已有 `FinRL/` 目录 |

**关键原则：**
- 先从 XGBoost/LightGBM 开始（数据量要求低、训练快、可解释性强）
- 使用滚动窗口训练（用过去 1 年数据预测下 1 周，滚动更新）
- 严格避免前视偏差（训练 T 日模型只能用 T 日及之前的数据）
- 先模拟后实盘（模拟盘跑 3 个月达标后再小资金实盘）

### 4.3 快速落地路线

1. **第 1 步**：用 XGBoost + Backtrader 跑通最小原型，计算核心评估指标
2. **第 2 步**：把预测结果和评估指标写入 Supabase `predictions` 表
3. **第 3 步**：尝试微软 Qlib，用其内置模型和回测系统优化
4. **第 4 步**：如果效果好，再尝试 FinRL 强化学习，加入反馈迭代机制

## 五、情感分析流水线（API 标注 + 本地分类器蒸馏）

### 5.1 方案选择

放弃 Cursor Skill 路线（Skill 只能在交互式对话中触发，无法在定时脚本中调用）。采用 **"API 标注 + 轻量级分类器蒸馏"** 方案：

| 方案 | 自动化 | 成本 | 效果 | 实现难度 | 推荐 |
|------|--------|------|------|----------|------|
| 纯 DeepSeek API | ✅ | 极低（≈¥1.8/年） | 最好 | 极低 | 短期首选 |
| API 标注 + FastEmbed + XGBoost | ✅ | 零 | 接近 API | 低 | 中长期目标 |
| 微调 BERT-tiny | ✅ | 零 | 略差 | 中 | 备选 |
| Cursor Skill | ❌ | 高 | 好 | 低 | 不适用 |

### 5.2 阶段一：纯 DeepSeek API（短期，现在～1 个月）

#### 成本计算

- DeepSeek V4 Flash：0.1 元/百万输入 token + 0.3 元/百万输出 token
- 单条新闻分析：输入≈200 token，输出≈50 token → ≈0.00005 元/条
- 每天 100 条新闻：≈0.005 元/天 → **一个月 ≈0.15 元，一年 ≈1.8 元**

#### 自动化流程

1. 从 Supabase `trend_news` 表读取未标注新闻（`sentiment_score IS NULL`）
2. 调用 DeepSeek API 分析（category/sentiment/impact/reason）
3. 将结果写回 `trend_news` 表，标记 `is_labeled_by_api = TRUE`
4. 交易日 18:00 后统一运行（与数据处理流水线合并），安全间隔 0.5-1.5 秒

#### 部署方式

- **本地**：Windows 任务计划程序，交易日 18:00 后触发（与基金数据处理合并为一条流水线）
- **云端**：GitHub Actions（API 脚本不会被限流，只有 AKShare 批量任务需要注意）

### 5.3 阶段二：蒸馏本地分类器（中期，1～3 个月）

当积累了 **500 条以上** API 标注数据后，训练本地分类器。

#### 技术选型

- **文本嵌入**：`BAAI/bge-small-zh-v1.5`（50MB，中文优化，CPU 推理极快）
- **分类器**：XGBoost（对表格数据效果最好，不需要调参）

#### 训练流程

1. 从 Supabase 读取 `is_labeled_by_api = TRUE` 的数据
2. 用 FastEmbed 生成文本嵌入（CPU 上每秒 100 条）
3. 训练两个 XGBoost 模型：情感分类器 + 新闻分类器
4. 保存为 `sentiment_model.pkl` + `category_model.pkl` + `label_encoder.pkl`

### 5.4 阶段三：自动切换与持续优化（长期，3 个月以上）

#### 自动切换逻辑

1. 先用本地模型预测
2. 如果情感置信度 >90% **且** 分类置信度 >85% → 使用本地结果
3. 否则 → 回退到 API，并将 API 结果加入训练集

#### 持续优化

- 每周自动用新标注数据重新训练一次模型
- 当本地模型准确率连续一周超过 92% → 可完全关闭 API，实现零成本
- 每季度手动抽查 100 条结果，修正错误标注

### 5.5 现成参考资源

- **fin-sentiment**（github.com/yearinfeng/fin-sentiment）— FastEmbed + XGBoost 中文金融情感分析，已预训练 10 万条金融新闻
- **Chinese-Finance-Text-Classification**（github.com/hellonlp/Chinese-Finance-Text-Classification）— 中文金融文本分类基准模型
- **TrendRadar 内置 AI 分析** — TrendRadar 本身支持 DeepSeek 分析，结果自动保存到 COS

    prediction_date DATE NOT NULL,

### 6.5 视图

```sql
-- 重点基金最新数据视图
CREATE VIEW focus_funds_latest AS
SELECT
    f.fund_code, f.fund_name, f.fund_type,
    fn.unit_nav, fn.daily_return,
    fs.total_score
FROM funds f
JOIN fund_nav fn ON f.fund_code = fn.fund_code
JOIN fund_scores fs ON f.fund_code = fs.fund_code
WHERE f.is_focus = TRUE
AND fn.nav_date = (SELECT MAX(nav_date) FROM fund_nav WHERE fund_code = f.fund_code)
AND fs.score_date = (SELECT MAX(score_date) FROM fund_scores WHERE fund_code = f.fund_code);
```

## 八、风险与注意事项

### 8.1 技术风险

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| AKShare 接口限流 | 批量数据获取被封 IP | 随机抖动间隔 + 分批处理 + 本地网络运行 |
| RSS 源失效 | 新闻数据断供 | 配置 10 个源 + AKShare 新闻接口备用 |
| DeepSeek API 不稳定 | 情感分析中断 | 实现重试机制 + 降级为中性标记 |
| Supabase 免费额度用尽 | 数据写入失败 | 监控用量，必要时升级套餐 |
| 数据处理/推送流程失败 | 信息断供未察觉 | 失败告警机制（详见 2.6），立即触发邮箱通知 |

### 8.2 数据风险

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 前视偏差 | 回测结果虚高 | 严格使用 T 日及之前的数据 |
| 标注质量 | 本地分类器学到错误模式 | 每季度抽查 100 条，修正错误标注 |
| 数据延迟 | 基金净值 T+1 才更新 | 预测目标设为 T+1，避免用未来数据 |

### 8.3 安全事项

| 事项 | 操作 | 紧急度 |
|------|------|--------|
| DeepSeek API Key 更换 | 在 DeepSeek 后台重新生成 | 高（已暴露） |
| Supabase access token 续期 | 2026-06-06 前在后台重新生成 | 中 |
| 腾讯云 COS 试用期 | 2026-11-07 前评估是否续费 | 低 |
| TrendRadar 试用期 | 每 7 天运行 "Check In" workflow | 低 |

### 8.4 避坑总则

1. **代码和 SQL 仅为参考** — 实际实现需根据 AKShare 接口返回格式、Supabase 实际行为进行调整
2. **先跑通再优化** — 不要一开始就追求完美，先用最简单的方案跑通全流程
3. **本地优先** — 批量数据获取在本地家庭网络运行，避免云服务器 IP 被限流
4. **定期备份** — Supabase 数据定期导出备份，防止意外丢失
